**(a) Onboarding checkpoint persistence — mechanism recommendation.** Persist **materialized results**, not a seed: `Math.random()` in `obStart()` (confirmed `js/app.js:2548-2550`) means the deck itself is non-reproducible from a seed alone, so true resume requires storing the actual built deck (ids + order), `admired`/`skipped` id arrays, `answers`, current `step`/`di`/`qi`, and `tones` — everything `ob` already holds in memory — not a re-derivable seed. Two storage targets, weighed:
- **sessionStorage, new key** (e.g. `pigment.onboarding.v1`): survives reload/back-forward within the tab, cleared on tab close. Matches the "in-progress, not yet a Passport" nature of onboarding — a half-finished quiz isn't taste data yet. Smallest blast radius: never touches `pigment.taste.v1`, so the existing additive-only rule (F4-i, unchanged) is untouched. **Recommended.**
- **localStorage, new additive key under the existing Passport namespace pattern** (not a `pigment.taste.v1` field — a sibling key): survives across sessions/devices-via-export, closer to what "recoverable" could mean generously, but conflates in-progress quiz state with the durable Passport artifact, and needs its own cleanup-on-`obFinish()`/`retake` discipline to avoid an orphaned half-quiz haunting a later session.
Recommend **sessionStorage**, additive key, written at every step transition (tone pick, each deck swipe, each question answer) and cleared on `obFinish()` or an explicit `retake` confirm. This satisfies AC7's "resume at the exact checkpoint with prior answers intact" without touching the Passport schema at all.

**(g) Google Fonts disposition — three options costed, smallest/reversible named.**
- **Option 1 — Retain with approval.** Zero code change; requires the qualified privacy determination + owner approval AC25 already demands. Smallest possible diff (none), but does not reduce the privacy surface.
- **Option 2 — Remove.** Delete `index.html:19-21`'s two `preconnect` links + the `css2?family=...` stylesheet link; fall back to system font stack. Smallest **code** diff (3 lines removed), fully reversible, zero residual third-party surface — but is a visual-direction change (Matisse's territory) since Playfair Display/Inter are load-bearing to the current type system.
- **Option 3 — Self-host.** Commit OFL-licensed Playfair Display + Inter `.woff2` subsets under a new `fonts/` directory, add `@font-face` rules to `css/styles.css`, remove the two Google hostnames entirely. Zero-dependency-compatible (no runtime third-party request) and preserves the visual direction, but **requires fetching the actual font binaries from a network source to commit them** — a step I am not authorized to perform in this advisory round (no URL fetches performed) and which itself would need the same "downloading files" scrutiny any file download gets. Also the largest diff of the three (new binary assets + `@font-face` block + `?v=` bump on `styles.css`).
**Smallest and most reversible: Option 1 (retain-with-approval)** — zero build risk, immediately available once the owner/privacy reviewer sign off. **Option 2 (remove)** is smallest in code terms but carries a visual-identity decision outside Dürer's authority. **Option 3 (self-host)** is the only option that satisfies both "zero-dependency" and "keep the current type system," but has the highest setup cost and a network-fetch precondition that must be authorized separately before I can execute it.
